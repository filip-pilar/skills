# Compress Without Changing Behavior

Compression reduces representation cost while preserving an accepted contract. It is not semantic optimization.

## Establish the preservation contract

Read the complete package and map activation and scope; inputs and inherited context; workflow, ordering, state, and re-entry; tools, approvals, and safeguards; outputs; and fallbacks or coverage limits. Define representative ordinary, state, failure, and safety scenarios. Keep this map internal unless the user asks for it.

Inspect known failures, real outputs, accepted evals, and the target for direct contradictions or material unprioritized tensions. If one exists, stop both edits and compression recommendations. Present two choices: switch to Improve, or knowingly preserve the issue and continue representation-only. The original compression request predates disclosure and is not acceptance of either choice. If the user chooses representation-only work, preserve and report the issue and do not describe the result as behaviorally optimized.

## Necessity test

For every instruction or resource, ask whether removal would materially increase the chance of wrong activation, lost capability, unsafe action, incorrect output, or failure in a realistic accepted scenario.

- Retain necessary behavior once, in the clearest location.
- Merge repeated constraints and overlapping phases.
- Remove generic model knowledge, non-actionable rationale, implementation history, oversized examples, and rigid taxonomies without demonstrated value.
- Delegate standard mechanics to authoritative scripts, references, specifications, tools, parent context, or invoked skills.
- Move conditional detail only when it reduces an actual loading path. Measure root and each selectively loaded path separately; the sum of every conditional reference is not normal runtime cost.
- Keep critical activation, routing, approval, safety, and stopping gates in the root when agents must see them before deciding whether to load a reference.
- Do not hide or duplicate bloat across files.
- Remove a resource only after establishing that no accepted behavior depends on it.

No change is a successful result when every instruction earns its place. Keep specificity proportional to fragility. Do not broaden or narrow scope, add features, reorder required operations, weaken safeguards, or erase meaningful edge cases. Use size as evidence, not a target; do not set an arbitrary goal or promise a reduction before inspecting the candidate diff.

## Parity check

Compare every contract item and replay the same applicable scenarios before and after. Phrase-presence checks are supporting evidence only. If an instruction or resource has uncertain equivalence, restore it or report the uncertainty instead of claiming regression-free compression.

Report actual before/after size for affected loading paths, meaningful removals or moves, preserved capabilities and safeguards, known semantic issues, validation performed, and the strongest parity evidence. Use one named counting method for baseline and candidate. When proposing a concrete candidate, name the representative scenarios still needed to verify it. Do not report speculative size ranges as results.
