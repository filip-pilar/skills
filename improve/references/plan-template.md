# Handoff plan template

Plans are for an executor with no advisor-session context. They require self-contained facts, exact boundaries, verification after each step, and specific STOP conditions.

Use `<plans-dir>/NNN-short-slug.md`, where `<plans-dir>` is the single directory selected for this run (`plans/` or the `advisor-plans/` fallback).

## Plan file

````markdown
# Plan NNN: <imperative outcome>

> **Executor instructions:** Follow each step and its verification. Stop instead
> of improvising when a STOP condition occurs. Update the plan index unless a
> reviewer explicitly says it owns that update.
>
> **Preflight:**
> ```bash
> test "$(git rev-parse HEAD)" = "<full-planned-SHA>"
> git diff --quiet <full-planned-SHA> -- <all-in-scope-paths>
> test -z "$(git ls-files --others --exclude-standard -- <all-in-scope-paths>)"
> ```
> Any failure means the plan or working tree is not represented by `Planned at`:
> STOP and report it.

## Status

- **Priority:** P1 | P2 | P3
- **Effort:** S | M | L
- **Risk:** LOW | MED | HIGH
- **Depends on:** `<plans-dir>/NNN-*.md` or none
- **Category:** bug | security | perf | tests | tech-debt | migration | dx | docs | direction
- **Planned at:** `<full commit SHA>`, `<YYYY-MM-DD>`
- **Issue:** `<URL>` only when published

## Why this matters

State the verified problem, concrete cost, and intended outcome in 2–5 sentences.

## Current state

- Name every relevant file and its role.
- Include short verified excerpts with `file:line` markers.
- State applicable constraints, vocabulary, and decided tradeoffs.
- Show one repository exemplar for conventions the implementation must match.

## Commands

| Purpose | Exact command | Expected result |
|---|---|---|
| Setup, if required | `<verified command>` | exit 0 |
| Focused tests | `<verified command>` | named tests pass |
| Full verification | `<verified command>` | exit 0; no new failures |

Use commands verified from this repository, never generic ecosystem guesses.

## Suggested executor tools

Optional. Name only tools or references that plausibly exist and materially help.

## Scope

**In scope — the only source paths that may change:**

- `path/file.ext`
- `path/new-test.ext` (create)

**Out of scope:**

- `path/related.ext` — why it must remain untouched.
- Public contracts or behaviors that must not change.

## Git workflow

- Use the reviewer-provided executor branch/worktree, or the repository's observed branch convention for manual execution.
- Match observed commit-message style; include one example when useful.
- Do not push or open a pull request without operator instruction.

## Steps

### 1. <imperative step>

Name exact files, symbols, and target behavior. Include a target code shape only when it prevents ambiguity.

**Verify:** `<command>` → `<expected result>`

### 2. <next step>

Keep steps independently verifiable and order them so the repository remains coherent.

**Verify:** `<command>` → `<expected result>`

## Test plan

- New or changed tests, their files, and named cases.
- Existing test used as the structural pattern.
- Focused and full verification commands with expected results.

## Done criteria

- [ ] Every step verification has its expected result.
- [ ] The regression/behavior is covered by meaningful assertions.
- [ ] Full repository checks pass, or pre-existing failures are evidenced and unchanged.
- [ ] No path outside Scope changed, including untracked files.
- [ ] The plan-index row is updated by its designated owner.

## STOP conditions

- Current code contradicts a quoted fact or excerpt.
- A required verification fails twice after one reasonable correction.
- The work requires an out-of-scope path or public-contract change.
- `<plan-specific load-bearing assumption>` is false.

## Maintenance notes

State what future work interacts with this change, what review should scrutinize, and what was deliberately deferred.
````

## Plan index

```markdown
# Implementation plans

Generated by `improve` on <date>. Read each plan fully, honor STOP conditions,
and execute in dependency order.

| Plan | Title | Priority | Effort | Depends on | Status |
|---|---|---|---|---|---|
| 001 | ... | P1 | S | — | TODO |

Statuses: TODO | IN PROGRESS | DONE | BLOCKED: reason | REJECTED: rationale

## Dependency notes

- 002 requires 001 because ...

## Findings considered and rejected

- Finding — not worth doing because ...
```

## Quality check

- Could a fresh executor succeed using only this file and the repository?
- Does every step name paths/symbols and end with an objective check?
- Do Scope paths exactly match preflight and done-criteria checks?
- Are STOP conditions specific rather than boilerplate?
- Are all excerpts independently verified and free of secret values?
- Does `Planned at` contain the full current commit SHA?
