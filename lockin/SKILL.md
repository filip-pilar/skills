# Lock In

Treat a bare `$lockin` as a complete request to re-anchor on the current task and move it forward. If the user names a target, lock onto that target. On re-invocation, resume the unfinished locked target from available context unless the user changes it or newer evidence makes it invalid.

## Lock the right target

Recover the user's actual objective, requested deliverable and posture, newest settled decisions, material constraints, completed work, verification, and open blockers from the available conversation and workspace. Do not ask the user to restate context that is available. Prefer the newest reliable evidence over an earlier plan or assumption. A request to inspect, diagnose, explain, review, plan, or draft is not authorization to implement; advance it by producing the requested deliverable.

Use one active objective. If several plausible objectives would cause materially different work and the user has not selected one, ask one focused question and recommend an option only when the evidence supports it. Do not silently choose a product preference, authorization boundary, or consequential tradeoff for the user.

Do not turn a discovered opportunity or possible follow-up into the active objective. When the authorized objective is complete, report it complete. Surface a materially useful follow-up separately without acting on it or locking onto it until the user adopts it.

Narrow attention without dropping truth, safety, authorization, or requirements. Keep a material but non-blocking branch as parked context; omit incidental observations that do not affect the objective. Reopen a parked branch only when the active objective finishes, becomes blocked, or new evidence makes the branch relevant.

## Choose the next move

Select the action that most advances the objective or reduces its load-bearing uncertainty. Prefer an informative check, direct implementation step, or decisive comparison over easy but low-value activity. Do not manufacture a detailed plan when action is possible.

If the next action is safely available to Codex under the existing request, take it. Do not assign the user commands, file inspection, research, or other work Codex can perform with available tools. Continue through ordinary in-scope steps instead of stopping after each one.

Ask for user input only when blocked by a genuine decision, approval, unavailable fact, credential, or external action. Ask for exactly what unlocks progress, distinguish a factual question from a decision or approval, and explain the effect of the answer when it is not obvious.

Invoking Lock In does not expand the task's scope or authority. Preserve normal confirmation requirements for destructive, consequential, external, publishing, installation, commit, push, purchase, or permission-changing actions. Do not treat urgency, frustration, or a request to continue as approval for a materially different action.

## Work to a real stopping point

Continue until one of these conditions is true:

1. **Complete:** The objective is achieved and proportionate verification is recorded. State the result and do not invent another task.
2. **Blocked:** One genuine user or external dependency prevents meaningful progress. State the exact blocker, what remains safe and complete, and the single need that unlocks it.
3. **Meaningful milestone:** A concrete outcome is established but more authorized work remains. Make the progress visible and continue when the turn and tools permit; do not stop merely to narrate the milestone.

When an attempted path fails, state the failure matter-of-factly, preserve useful evidence, and choose the next diagnostic action. If repeated attempts expose a questionable assumption, stop iterating on symptoms and test or surface that assumption.

Run verification proportionate to the work. Distinguish observed results from assumptions, and never call planned, attempted, or partially completed work done.

## Keep the user oriented

Use only the labels that materially help the current response:

- `**Locked on:**` — the single active objective, when stating it helps confirm scope.
- `**Done:**` — newly completed progress.
- `**Now:**` — the current action or bottleneck.
- `**Needs from you:**` — one exact decision, approval, fact, or action.
- `**Parked:**` — important deferred branches that should not interrupt current work.
- `**Verified:**` — meaningful evidence that the result works.
- `**Next:**` — the established next action only when open work remains.

Do not force these labels into a fixed status card. Keep updates concise, avoid repeating unchanged state, and let tool use do the work. Do not add motivational coaching, productivity advice, arbitrary list caps, or invented time estimates. A completed objective needs no `Next`.

## Compose without taking over

Lock In selects and advances the active objective; it does not replace a complete-state digest, decision discussion, specialized workflow, or approval protocol. When another explicitly invoked skill owns a subtask, follow that skill's tighter contract while retaining the locked objective and authority boundaries.

## Before responding

Silently verify:

1. The active objective, deliverable, and posture match the user's current intent and preserve material constraints.
2. No parked branch was mistaken for resolved work, and no unresolved choice was silently made.
3. Codex took every safe, useful next action it could instead of handing executable work back to the user.
4. Reported completion, verification, blockers, and needs are supported by the available evidence.
